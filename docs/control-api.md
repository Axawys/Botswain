# Botswain Control API — v0

Единый контракт между агентом (`/agent`) и клиентом (`/app`). Клиент общается
с агентом **только** через этот API. Прямой шелл произвольных команд по SSH из
GUI запрещён: SSH используется ровно для двух вещей — (1) первичный bootstrap
сервера, (2) поднятие port-forward туннеля к агенту.

Этот документ — источник правды. Он пишется раньше реализации; и агент, и клиент
следуют ему.

---

## Транспорт и доступ

- Агент биндится на `127.0.0.1:<port>` (по умолчанию `8080`, см. флаг `--port`).
  Публично в сеть агент **не** торчит.
- На VPS порт агента пробрасывается на loopback хоста
  (`docker run -p 127.0.0.1:PORT:PORT ...`), клиент дотягивается через SSH
  port-forward (локальный порт клиента → `127.0.0.1:PORT` на VPS).
- В локальном режиме — тот же агент на `127.0.0.1`, тот же API, отличается
  только транспорт (без туннеля).
- **Аутентификация в v0 = сам туннель.** Кто смог поднять SSH-сессию, тот и
  имеет доступ к API. Отдельного токена в M1 нет.
  - _Зарезервировано на будущее:_ заголовок `Authorization: Bearer <token>`.
    В v0 агент его игнорирует.
- Базовый префикс всех путей: `/v0`.
- Кодировка — UTF-8. Тело запросов/ответов — `application/json`.

---

## Формат успешного ответа

Хендлеры возвращают ресурс **напрямую**, без обёртки-конверта. Семантику несёт
HTTP-статус.

```http
GET /v0/health  →  200 OK
Content-Type: application/json; charset=utf-8
```

```json
{
  "status": "ok",
  "version": "0.1.0",
  "commit": "09f9e35",
  "uptime_seconds": 42,
  "time": "2026-07-01T12:00:00Z"
}
```

Поле `status`:

- `"ok"` — агент готов обслуживать запросы.
- `"not_ready"` — процесс жив, но критичная зависимость ещё недоступна
  (в будущем, например, `/var/run/docker.sock`). В v0 всегда `"ok"`.

Добавление новых полей в ответ считается **несовместимо-безопасным**: клиент
обязан игнорировать незнакомые поля.

## Формат ошибки

Любой ответ не-2xx — единый конверт:

```json
{
  "error": {
    "code": "docker_unavailable",
    "message": "cannot reach docker daemon via /var/run/docker.sock",
    "details": {}
  }
}
```

- `code` — стабильный машиночитаемый строковый код (`snake_case`). Клиент
  ветвит логику по нему.
- `message` — человекочитаемо, для логов и диагностики. **Не** для парсинга.
- `details` — опциональный объект с дополнительным контекстом. Может отсутствовать.

### Коды ошибок v0 (стартовый набор)

| HTTP | `code`                | Когда                                        |
|------|-----------------------|----------------------------------------------|
| 404  | `not_found`           | неизвестный путь                             |
| 405  | `method_not_allowed`  | путь есть, метод неверный                    |
| 500  | `internal`            | необработанная внутренняя ошибка             |
| 503  | `not_ready`           | агент поднялся, но ещё не готов обслуживать   |

Коды, относящиеся к ботам (M2), перечислены в разделе «Боты».

## Эндпоинты v0

### `GET /v0/health`

Проверка живости и готовности агента. Дёшев, без побочных эффектов — рассчитан
на health-check при bootstrap и на периодический polling состояния туннеля.

- `200 OK` + тело со `status: "ok"` — агент готов.
- `503 Service Unavailable` + `{"error":{"code":"not_ready"}}` — процесс жив,
  но критичная зависимость недоступна. Начиная с M2 сюда попадает случай, когда
  агент не может достучаться до `/var/run/docker.sock`.

Пример:

```bash
curl -s http://127.0.0.1:8080/v0/health
```

---

## Последовательность bootstrap + health-check (M1)

1. Клиент по SSH проверяет наличие Docker (`docker --version`). Если нет —
   установка (в M1 — TODO-заглушка, но проверка выполняется).
2. Клиент по SSH запускает агента:
   `docker run -d --restart unless-stopped -p 127.0.0.1:PORT:PORT ... botswain-agent`.
3. Клиент поднимает SSH port-forward: `127.0.0.1:LOCAL → 127.0.0.1:PORT` на VPS.
4. Клиент опрашивает `GET /v0/health` с экспоненциальным backoff
   (например: 500ms → удвоение → cap 5s, общий дедлайн ~30s).
5. Первый `200 {"status":"ok"}` → UI показывает «агент жив».

Сетевой слой клиента обязан переживать разрыв туннеля: при обрыве — повторное
поднятие туннеля и возобновление polling'а с backoff.

---

## Версионирование и совместимость

- Ломающие изменения → новый префикс пути (`/v1`), при этом `/v0` живёт
  параллельно до явной депрекации.
- Добавление полей в ответы — не ломающее изменение; клиент игнорирует
  незнакомые поля.

## Боты (M2)

Управление жизненным циклом ботов. Каждый бот — соседний Docker-контейнер из
базового образа `python:3.12-slim`, с автоперезапуском и лимитами ресурсов.
Источник правды о ботах — сами контейнеры: агент не ведёт отдельную БД, а хранит
метаданные в labels контейнера (`botswain.*`). Подробности рантайма — в
[architecture.md](architecture.md).

Код бота доставляется **через агента**: клиент присылает архив, агент
распаковывает его в per-bot volume и ставит зависимости. Прямого доступа к
файловой системе сервера у клиента нет.

### Модель ресурса `Bot`

```json
{
  "id": "b3f1c2a4",
  "name": "my-echo-bot",
  "entrypoint": "main.py",
  "status": "running",
  "limits": { "memory_mb": 256, "cpus": 0.5 },
  "image": "python:3.12-slim",
  "created_at": "2026-07-01T12:00:00Z"
}
```

- `id` — короткий идентификатор бота; он же в label `botswain.bot.id` и в имени
  контейнера `botswain-bot-<id>`.
- `status` — зеркалит состояние Docker-контейнера:
  `created` | `running` | `restarting` | `paused` | `exited` | `dead`.
- `limits.memory_mb` — лимит памяти в мегабайтах; `limits.cpus` — доля CPU
  (например `0.5` = половина ядра).

### Эндпоинты

| Метод   | Путь                    | Назначение |
|---------|-------------------------|------------|
| `POST`  | `/v0/bots`              | Создать и запустить бота. |
| `GET`   | `/v0/bots`              | Список ботов. |
| `GET`   | `/v0/bots/{id}`         | Один бот. |
| `POST`  | `/v0/bots/{id}/start`   | Запустить остановленного. |
| `POST`  | `/v0/bots/{id}/stop`    | Остановить. |
| `POST`  | `/v0/bots/{id}/restart` | Перезапустить. |
| `DELETE`| `/v0/bots/{id}`         | Удалить контейнер и volume бота. |
| `GET`   | `/v0/bots/{id}/metrics` | Снапшот CPU/RAM (M3, см. «Метрики и логи»). |
| `GET`   | `/v0/bots/{id}/logs`    | Стриминг логов по WebSocket (M3). |

Обновление кода/спека в M2 = `DELETE` + `POST` (отдельный `PUT` — позже).

#### `POST /v0/bots`

`Content-Type: multipart/form-data` с двумя частями:

- `spec` — `application/json`:
  ```json
  {
    "name": "my-echo-bot",
    "entrypoint": "main.py",
    "limits": { "memory_mb": 256, "cpus": 0.5 }
  }
  ```
- `code` — архив `.tar.gz`. В корне архива лежит файл `entrypoint` и, опционально,
  `requirements.txt` (зависимости ставятся при создании).

Ответ: `201 Created` + объект `Bot`.

#### Действия жизненного цикла

`start` / `stop` / `restart` возвращают `200 OK` + актуальный объект `Bot`.
`DELETE` возвращает `204 No Content`.

### Коды ошибок ботов

| HTTP | `code`                  | Когда |
|------|-------------------------|-------|
| 404  | `bot_not_found`         | нет бота с таким id |
| 409  | `bot_name_conflict`     | имя уже занято другим ботом |
| 400  | `invalid_spec`          | часть `spec` не парсится или нет обязательных полей |
| 400  | `invalid_archive`       | часть `code` не читается как `.tar.gz` |
| 422  | `entrypoint_not_found`  | указанного `entrypoint` нет в архиве |
| 500  | `install_failed`        | не удалось поставить зависимости (`pip`) |
| 503  | `docker_unavailable`    | агент не достучался до `/var/run/docker.sock` |

## Метрики и логи (M3)

### `GET /v0/bots/{id}/metrics`

Мгновенный снапшот потребления ресурсов ботом. Состояния на агенте нет: клиент
опрашивает эндпоинт с нужным интервалом (обычно раз в 1–2 с) и рисует графики
сам. Агент считает CPU% по двум подряд замерам Docker-статистики, поэтому ответ
приходит примерно через секунду.

`200 OK`:

```json
{
  "cpu_percent": 12.5,
  "memory_used_mb": 45.2,
  "memory_limit_mb": 128.0,
  "memory_percent": 35.3,
  "disk_used_mb": 210.4
}
```

- `cpu_percent` — доля CPU в процентах; `100` = одно полное ядро, может быть
  больше при нескольких ядрах.
- `memory_used_mb` / `memory_limit_mb` — использование и лимит памяти в МБ.
- `memory_percent` — `memory_used_mb / memory_limit_mb * 100`.
- `disk_used_mb` — место на диске, занятое контейнером бота (rootfs: слои образа
  + writable-слой), в МБ.

Ошибки: `404 bot_not_found`, `503 docker_unavailable`.

### `GET /v0/bots/{id}/logs` — WebSocket

Стриминг логов бота в реальном времени. Клиент открывает **WebSocket** к этому
пути (через тот же SSH-туннель). Аутентификация — сам туннель, как и для HTTP.

- Сервер сначала отдаёт «хвост» последних строк, затем продолжает слать новые по
  мере появления (`follow`).
- Каждое сообщение — **текстовый** фрейм с сырым выводом бота (stdout+stderr,
  демультиплексированные). Фрейм может содержать несколько строк или часть
  строки — клиент просто дописывает текст в консоль.
- Соединение живёт, пока клиент его не закроет или бот не будет удалён.

Ошибки до апгрейда: `404 bot_not_found`, `503 docker_unavailable`.
